# SPEC-053 — Savia Capability Map (.scm)

**Status:** Draft | **Author:** Savia | **Date:** 2026-03-29

## Problem

Savia has 500+ commands, 88 skills, 47 agents, and 245 scripts. When a user
speaks naturally ("review my PR", "check security"), Savia must resolve which
resource to invoke. Existing discovery mechanisms (tool-discovery, skill-auto-activation,
NL-command-resolution, smart-routing) each cover part of the problem but lack a
unified, lightweight index across ALL resource types.

## Solution

A Savia Capability Map (`.scm`) — an auto-generated, intent-indexed catalog of
every invocable resource. Like `.acm` maps code for agents, `.scm` maps Savia's
own capabilities so she knows what she can do and when to use each resource.

## Design Principles

1. **Progressive loading**: L0 index (~400 tokens) always available; L1 category
   summaries on demand; L2 full docs only during execution
2. **Intent-based**: indexed by what the user wants, not by file path
3. **Self-maintaining**: auto-generated by `scripts/generate-capability-map.sh`
4. **Lightweight**: INDEX.scm fits in ~500 tokens max

## Architecture

### L0: INDEX.scm (~400 tokens)

One-line-per-resource format, sorted by category:

```
[category] name — intent_keywords — type:relative_path
```

Resource types: `cmd`, `skill`, `agent`, `script`.
Categories: quality, development, planning, analysis, memory, communication, governance.

### L1: .scm/categories/{category}.scm

Per-category file with name + description (from frontmatter) for each resource.
Loaded when Savia needs to pick within a category. ~50-100 tokens per category.

### L2: Full resource file

The actual .md or .sh file. Loaded only during execution. Discarded after.

## Category Taxonomy (7 categories)

| Category | User intents | Prefixes |
|----------|-------------|----------|
| quality | test, review, audit, security, lint, coverage | test-, pr-, security-, a11y- |
| development | implement, code, build, deploy, spec, design | spec-, dev-, arch-, code-, pipeline- |
| planning | sprint, capacity, estimate, decompose, assign | sprint-, pbi-, capacity-, project- |
| analysis | trace, metrics, performance, debt, risk | report-, dora-, debt-, risk-, perf- |
| memory | recall, save, search, consolidate, context | memory-, context-, nl- |
| communication | notify, message, digest, meeting, inbox | msg-, notify-, meeting-, inbox- |
| governance | compliance, policy, risk, equality, aepd | compliance-, governance-, aepd-, bias- |

## Generator Script

`scripts/generate-capability-map.sh` scans four directories:

1. `.claude/commands/*.md` — extracts `name` and `description` from YAML frontmatter
2. `.claude/skills/*/SKILL.md` — extracts `name`, `description`, `tags` from frontmatter
3. `.claude/agents/*.md` — extracts `name` and `description` from frontmatter
4. `scripts/*.sh` — extracts description from first comment line after shebang

Category assignment uses prefix matching (from tool-discovery.md groups) with
fallback to keyword matching against intent keywords per category.

## Integration

- **NL-command-resolution**: uses INDEX.scm as fast lookup before intent-catalog
- **skill-auto-activation**: L0 data replaces scanning all SKILL.md frontmatters
- **smart-routing**: category files (.scm/categories/) replace ad-hoc grouping
- **session-init**: loads INDEX.scm header (resource counts) as context orientation

## File Layout

```
.scm/
├── INDEX.scm                    ← L0: one line per resource (~400 tokens)
└── categories/
    ├── quality.scm              ← L1: name + description per resource
    ├── development.scm
    ├── planning.scm
    ├── analysis.scm
    ├── memory.scm
    ├── communication.scm
    └── governance.scm
```

## Regeneration

Run `bash scripts/generate-capability-map.sh` after adding/removing resources.
The script is idempotent — safe to run anytime. Output is deterministic (sorted).

## Constraints

- INDEX.scm max 500 tokens (~250 lines with abbreviation)
- Category files max 150 lines each
- Git-tracked (N1 public knowledge about workspace structure)
- No PII, no project data, no secrets — only resource metadata

## Relationship to Existing Systems

| System | Role after .scm |
|--------|----------------|
| tool-discovery.md | Defines category taxonomy (source of truth for groups) |
| skill-auto-activation | Uses L0 for scoring instead of scanning frontmatters |
| smart-routing | Uses .scm/categories/ for category-scoped search |
| NL-command-resolution | Uses INDEX.scm as fast-path before intent-catalog |
| .acm (Agent Code Maps) | Maps code structure; .scm maps capability structure |

## Success Metrics

- INDEX.scm < 500 tokens (measured by `wc -w / 4`)
- Generator completes in < 5s on current workspace
- 7 category files generated, each < 150 lines
- All existing commands, skills, agents indexed
