## 1. Philosophy and Culture

Go is a language of **pragmatic minimalism**. The culture rejects complexity
that does not earn its keep. There is no inheritance, no exceptions, no
generics-for-everything. Every feature that exists was added because the
cost of NOT having it exceeded the cost of the complexity it introduced.

This model exists because Go projects fail in a specific way: developers
coming from Java, C#, or TypeScript impose patterns that fight the
language. They create interface hierarchies nobody asked for, abstract
factories for things that need a constructor, and "clean architecture"
layers where a flat package would suffice. The result is Go code that
reads like Java with worse syntax.

**The Go way**: start simple. A single `main.go` is fine for a CLI tool.
A flat package is fine for a small library. Add structure ONLY when you
feel the pain of not having it. The pain is the signal — not a blog post,
not a template repository, not a conference talk.

### Core tenets

1. **Clear is better than clever.** A 20-line function with explicit
   error handling beats a 5-line function using reflection and generics.
2. **A little copying is better than a little dependency.** Do not import
   a package to save 10 lines of code you could write yourself.
3. **Interfaces belong to consumers, not producers.** Define interfaces
   where they are used, not where they are implemented. Accept interfaces,
   return structs.
4. **Make the zero value useful.** Design types so that their zero value
   is a valid, usable state — `sync.Mutex{}`, `bytes.Buffer{}`.
5. **Errors are values, not exceptions.** Handle them explicitly at every
   call site. Wrapping with `fmt.Errorf("context: %w", err)` builds a
   diagnostic chain.
6. **Concurrency is a tool, not a goal.** Use goroutines when you have
   independent work to do. Do not use channels when a mutex suffices.

### When this model IS appropriate

- REST APIs (stdlib, gin, chi) of any size
- CLI tools (cobra, bare flag package)
- Microservices communicating via gRPC or HTTP
- Infrastructure tools (Terraform providers, Kubernetes operators)
- Libraries and SDK packages published to pkg.go.dev
- High-throughput data processing pipelines

### When this model is NOT appropriate

- GUI desktop applications (Go has no mature UI toolkit)
- iOS/Android mobile apps (use Kotlin/Swift)
- Frontend web UIs (use TypeScript)
- Machine learning model training (use Python)

### Trade-offs accepted

- **Verbose error handling**: every function call that can fail needs an
  `if err != nil` block. This is a feature: errors are visible, not hidden.
- **No inheritance**: composition via embedding. No polymorphic hierarchies.
- **Limited generics**: Go generics are intentionally constrained compared
  to Rust/C++. Use them for containers and algorithms, not for DI frameworks.
- **No implicit behavior**: no constructors, no destructors, no operator
  overloading. Everything is explicit. Code reads linearly.

## 2. Architecture Principles

### The "Flat-Then-Layered" Approach

Go projects should grow their structure organically:

**Stage 1 — Flat** (0-2K LOC, 1-3 developers):
```
main.go
handler.go
storage.go
go.mod
```
Everything in package `main`. No layers. No interfaces. Direct function calls.

**Stage 2 — Internal packages** (2-10K LOC, 2-5 developers):
```
cmd/myapp/main.go
internal/handler/
internal/storage/
internal/domain/
```
Split when packages have distinct responsibilities. `internal/` enforces
encapsulation at the compiler level.

**Stage 3 — Ports and Adapters** (10K+ LOC, 5+ developers):
```
cmd/myapp/main.go
internal/app/         <- application services (use cases)
internal/domain/      <- entities, value objects, domain errors
internal/port/        <- interfaces (defined by consumers!)
internal/adapter/     <- implementations (HTTP, DB, queue)
```
Only add this when you genuinely need to swap implementations or when
testing requires it. Most Go projects never reach Stage 3.

### Per-Application-Type Architecture

#### REST API (stdlib / gin / chi)

```
cmd/api/main.go              <- wiring, config, server startup
internal/
  handler/                   <- HTTP handlers (thin: decode, call service, encode)
  service/                   <- business logic (no HTTP awareness)
  storage/                   <- database access (sqlc-generated or manual)
  middleware/                <- auth, logging, recovery, CORS
  model/                     <- domain types, request/response types
```

Handlers are thin. They decode the request, call a service, encode the
response. A handler NEVER contains business logic. If your handler has
an `if` that is not about HTTP (status codes, headers), move it to a service.

#### CLI Tool (cobra)

```
cmd/mytool/main.go           <- cobra root command
internal/
  cmd/                       <- subcommand definitions
  runner/                    <- business logic behind each command
  output/                    <- formatters (table, JSON, YAML)
```

Cobra is the standard. For simple CLIs with 1-3 flags, use the stdlib
`flag` package directly. Do not import cobra for a 50-line tool.

#### Microservice

Same as REST API, plus:
```
internal/
  event/                     <- message producers/consumers
  grpc/                      <- gRPC server and client wrappers
  health/                    <- readiness/liveness probes
```

#### Library / SDK

```
mylib.go                     <- public API (package-level functions or types)
mylib_test.go
options.go                   <- functional options if constructor is complex
internal/                    <- private implementation details
examples/                    <- runnable examples (go test -run Example)
```

Libraries MUST have zero dependencies beyond the standard library where
possible. Every dependency you add is a dependency your users inherit.

### Dependency Direction

```
cmd/         -> internal/    (wiring, never business logic)
handler/     -> service/     (handlers depend on services)
service/     -> storage/     (services depend on storage INTERFACES)
storage/     -> database     (implementations depend on infrastructure)
```

Interfaces live where they are consumed. `service/` defines a
`UserStore` interface. `storage/` implements it. `service/` never
imports `storage/` — it only knows the interface.

```go
// internal/service/user.go
type UserStore interface {
    GetByID(ctx context.Context, id int64) (User, error)
    Create(ctx context.Context, u User) error
}

type UserService struct {
    store UserStore  // injected via constructor
    log   *slog.Logger
}
```

## 4. Code Patterns

### Error Handling — The Go Way

Errors are values. Handle them explicitly. Wrap with context.

```go
// Wrapping errors with context (since Go 1.13)
user, err := s.store.GetByID(ctx, id)
if err != nil {
    return User{}, fmt.Errorf("get user %d: %w", id, err)
}

// Checking error types with errors.Is (sentinel errors)
if errors.Is(err, storage.ErrNotFound) {
    return http.StatusNotFound, nil
}

// Checking error types with errors.As (custom error types)
var validErr *ValidationError
if errors.As(err, &validErr) {
    return http.StatusBadRequest, validErr.Fields()
}
```

**Sentinel errors** for known failure modes:
```go
// internal/model/errors.go
var (
    ErrNotFound    = errors.New("not found")
    ErrConflict    = errors.New("conflict")
    ErrForbidden   = errors.New("forbidden")
    ErrInvalidInput = errors.New("invalid input")
)
```

**Custom error types** when you need structured information:
```go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation: %s — %s", e.Field, e.Message)
}
```

### Structured Logging with slog (Go 1.21+)

slog is the standard. Do not import zerolog, zap, or logrus for new projects.

```go
// Setup in main.go
logger := slog.New(slog.NewJSONHandler(os.Stdout, &slog.HandlerOptions{
    Level: slog.LevelInfo,
}))
slog.SetDefault(logger)

// Usage in services — pass logger explicitly
func NewUserService(store UserStore, log *slog.Logger) *UserService {
    return &UserService{store: store, log: log.With("component", "user-service")}
}

func (s *UserService) Create(ctx context.Context, u User) error {
    s.log.InfoContext(ctx, "creating user", "email", u.Email)
    if err := s.store.Create(ctx, u); err != nil {
        s.log.ErrorContext(ctx, "failed to create user",
            "email", u.Email, "error", err)
        return fmt.Errorf("create user: %w", err)
    }
    return nil
}
```

Use `slog.With()` for request-scoped attributes (request ID, user ID)
injected via middleware.

### Context Propagation

Every function that does I/O or may be cancelled takes `context.Context`
as its first parameter. No exceptions.

```go
func (s *UserService) GetByID(ctx context.Context, id int64) (User, error) {
    // ctx carries cancellation, deadlines, and request-scoped values
    return s.store.GetByID(ctx, id)
}
```

### Goroutines and Channels

Use goroutines for independent, concurrent work. Always handle cleanup.

```go
// CORRECT: bounded concurrency with errgroup
func (s *Service) ProcessBatch(ctx context.Context, ids []int64) error {
    g, ctx := errgroup.WithContext(ctx)
    g.SetLimit(10) // max 10 concurrent goroutines

    for _, id := range ids {
        g.Go(func() error {
            return s.processOne(ctx, id)
        })
    }
    return g.Wait()
}

// CORRECT: producer-consumer with channels
func generate(ctx context.Context) <-chan Item {
    ch := make(chan Item)
    go func() {
        defer close(ch) // ALWAYS close channels from the sender
        for item := range source {
            select {
            case ch <- item:
            case <-ctx.Done():
                return
            }
        }
    }()
    return ch
}
```

### Generics (Go 1.18+) — Use Sparingly

Generics are for **containers, algorithms, and utility functions**. Not for
business logic. Not for dependency injection frameworks.

```go
// GOOD: generic utility function
func Map[T, U any](s []T, f func(T) U) []U {
    result := make([]U, len(s))
    for i, v := range s {
        result[i] = f(v)
    }
    return result
}

// GOOD: generic result type
type Result[T any] struct {
    Value T
    Err   error
}

func NewResult[T any](v T, err error) Result[T] {
    return Result[T]{Value: v, Err: err}
}
```

### Range-Over-Func Iterators (Go 1.23+)

Go 1.23 introduced range-over-function types, enabling custom iteration.

```go
// Iterator that yields filtered users
func ActiveUsers(users []User) iter.Seq[User] {
    return func(yield func(User) bool) {
        for _, u := range users {
            if u.Active {
                if !yield(u) {
                    return
                }
            }
        }
    }
}

// Usage: range over the iterator like a slice
for user := range ActiveUsers(allUsers) {
    fmt.Println(user.Name)
}
```

Use iterators when returning a full slice would be wasteful (large datasets,
lazy evaluation, streaming). For small collections, a plain slice is simpler.

### Functional Options for Configuration

```go
type ServerOption func(*Server)

func WithPort(port int) ServerOption {
    return func(s *Server) { s.port = port }
}

func WithLogger(l *slog.Logger) ServerOption {
    return func(s *Server) { s.log = l }
}

func NewServer(opts ...ServerOption) *Server {
    s := &Server{port: 8080, log: slog.Default()} // sane defaults
    for _, opt := range opts {
        opt(s)
    }
    return s
}
```

### Table-Driven Tests

The canonical Go testing pattern. Every test function should use this.

```go
func TestValidateEmail(t *testing.T) {
    tests := []struct {
        name  string
        email string
        want  bool
    }{
        {"valid email", "user@example.com", true},
        {"missing @", "userexample.com", false},
        {"empty string", "", false},
        {"unicode local", "user@example.es", true},
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := ValidateEmail(tt.email)
            if got != tt.want {
                t.Errorf("ValidateEmail(%q) = %v, want %v",
                    tt.email, got, tt.want)
            }
        })
    }
}
```

## 5. Testing and Quality

### Test Organization

```
mypackage/
  user.go
  user_test.go              <- unit tests (same package, white-box)
  user_integration_test.go  <- integration tests (build tag: //go:build integration)
  testdata/                 <- fixtures, golden files
```

Use build tags to separate fast unit tests from slow integration tests:

```go
//go:build integration

package storage_test

func TestPostgresUserStore(t *testing.T) {
    // uses real database via testcontainers
}
```

### Testing with testify

testify provides assertions and mocks. Use it for readability.

```go
import (
    "testing"
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestCreateUser(t *testing.T) {
    svc := NewUserService(mockStore, slog.Default())

    user, err := svc.Create(ctx, validInput)

    require.NoError(t, err)          // fatal on failure — stops test
    assert.Equal(t, "alice", user.Name)  // non-fatal — continues
    assert.NotZero(t, user.ID)
}
```

Use `require` for preconditions (stop if failed). Use `assert` for
verifications (report all failures).

### Integration Tests with testcontainers-go

Real databases in tests. No mocks for storage layer.

```go
//go:build integration

func TestUserStorePostgres(t *testing.T) {
    ctx := context.Background()

    pgContainer, err := postgres.Run(ctx,
        "postgres:16-alpine",
        postgres.WithDatabase("testdb"),
        postgres.WithUsername("test"),
        postgres.WithPassword("test"),
        testcontainers.WithWaitStrategy(
            wait.ForLog("ready to accept connections").
                WithOccurrence(2).WithStartupTimeout(5*time.Second),
        ),
    )
    require.NoError(t, err)
    t.Cleanup(func() { pgContainer.Terminate(ctx) })

    connStr, err := pgContainer.ConnectionString(ctx, "sslmode=disable")
    require.NoError(t, err)

    store, err := NewPostgresUserStore(ctx, connStr)
    require.NoError(t, err)

    // Now test against a real PostgreSQL
    err = store.Create(ctx, User{Name: "alice", Email: "alice@test.com"})
    require.NoError(t, err)

    user, err := store.GetByID(ctx, 1)
    require.NoError(t, err)
    assert.Equal(t, "alice", user.Name)
}
```

## 6. Security and Data Sovereignty

### Input Validation

Validate at the handler boundary. Never trust user input.

```go
func (h *Handler) CreateUser(w http.ResponseWriter, r *http.Request) {
    var req CreateUserRequest
    if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
        http.Error(w, "invalid JSON", http.StatusBadRequest)
        return
    }
    if err := req.Validate(); err != nil {
        http.Error(w, err.Error(), http.StatusBadRequest)
        return
    }
    // proceed with validated data
}
```

### SQL Injection Prevention with sqlc

sqlc generates type-safe Go code from SQL queries. No string concatenation.
No ORM magic. Plain SQL with compile-time safety.

```sql
-- db/queries/user.sql
-- name: GetUserByID :one
SELECT id, name, email, created_at
FROM users
WHERE id = $1;

-- name: CreateUser :one
INSERT INTO users (name, email)
VALUES ($1, $2)
RETURNING id, name, email, created_at;
```

sqlc generates Go structs and functions with parameterized queries:
```go
// generated — never edit
func (q *Queries) GetUserByID(ctx context.Context, id int64) (User, error) {
    row := q.db.QueryRowContext(ctx, getUserByID, id)
    // ...
}
```

### Crypto and TLS

```go
// HTTPS server with modern TLS config
srv := &http.Server{
    Addr:    ":8443",
    Handler: router,
    TLSConfig: &tls.Config{
        MinVersion: tls.VersionTLS13,
    },
    ReadTimeout:  5 * time.Second,
    WriteTimeout: 10 * time.Second,
    IdleTimeout:  120 * time.Second,
}
```

### Security Checklist

- All HTTP handlers set `Content-Type` explicitly
- CORS configured per-origin, never `*` in production
- JWT validation uses `crypto/ecdsa` or `crypto/ed25519`, never HS256 with a
  shared secret in environment variables
- Rate limiting per IP (stdlib `rate.Limiter` or middleware)
- Secrets loaded from environment or vault, never from config files in repo
- Dependencies scanned: `govulncheck ./...` in CI
- No `unsafe` package usage unless benchmarked and justified

### Savia Shield Integration

Go services that process client data (N4) integrate with Savia Shield:
- Hook `data-sovereignty-gate.sh` scans Go source for hardcoded credentials
- sqlc queries are inherently parameterized (no injection vector)
- slog structured logging NEVER logs PII — use `slog.String("user_id", id)`
  not `slog.String("user_email", email)`

## 7. DevOps and Operations

### Build

```makefile
# Makefile
.PHONY: build test lint run

build:
	CGO_ENABLED=0 go build -ldflags="-s -w" -o bin/api ./cmd/api

test:
	go test -race -count=1 ./...

test-integration:
	go test -race -count=1 -tags=integration ./...

lint:
	golangci-lint run ./...

run:
	go run ./cmd/api
```

`CGO_ENABLED=0` produces a statically linked binary. `-ldflags="-s -w"`
strips debug symbols for smaller binaries.

### Docker — Scratch Image

```dockerfile
# Build stage
FROM golang:1.23-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o /api ./cmd/api

# Runtime stage — scratch = 0 bytes base
FROM scratch
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/
COPY --from=builder /api /api
EXPOSE 8080
ENTRYPOINT ["/api"]
```

Result: a Docker image under 15MB containing a single static binary.

## 8. Anti-Patterns and Guardrails

### 15 DOs

| # | DO | Rationale |
|---|-----|-----------|
| 1 | Handle every error explicitly | Errors are values; ignoring them hides bugs |
| 2 | Use `context.Context` as first parameter for I/O | Enables cancellation, deadlines, tracing |
| 3 | Define interfaces at the consumer, not the producer | Decouples without premature abstraction |
| 4 | Use `internal/` to enforce encapsulation | Compiler-enforced; no runtime cost |
| 5 | Write table-driven tests for every function | Systematic, easy to extend, canonical Go |
| 6 | Use `slog` for structured logging | Standard library since 1.21; machine-parseable |
| 7 | Use `errgroup` for bounded concurrent work | Handles errors, cancellation, and limits |
| 8 | Use `sqlc` for database queries | Compile-time SQL validation, no injection |
| 9 | Return structs, accept interfaces | Keeps APIs flexible without over-abstracting |
| 10 | Use functional options for complex constructors | Extensible, backward-compatible, self-documenting |
| 11 | Set timeouts on all HTTP clients and servers | Prevent resource leaks and hanging connections |
| 12 | Use `go vet`, `staticcheck`, and `golangci-lint` | Catch bugs before they reach review |
| 13 | Use build tags for integration tests | Keep `go test ./...` fast (< 30 seconds) |
| 14 | Preallocate slices when size is known | `make([]T, 0, n)` avoids repeated allocation |
| 15 | Use `defer` for cleanup (close files, unlock mutexes) | Guaranteed execution, even on panic |

### 15 DONTs

| # | DONT | Why it fails in Go |
|---|------|-------------------|
| 1 | Abuse `init()` functions | Hidden side effects, untestable, order-dependent |
| 2 | Create interfaces before you have 2+ implementations | Interface pollution; Go interfaces are implicit |
| 3 | Use `interface{}` / `any` when a concrete type works | Loses type safety; generics exist for containers |
| 4 | Launch goroutines without a cancellation mechanism | Goroutine leaks crash production at 3AM |
| 5 | Use `panic` for expected errors | Panic is for programmer bugs, not user input |
| 6 | Import a package to save 10 lines of code | Dependency = maintenance burden for all users |
| 7 | Create a `utils/` or `helpers/` package | Junk drawer; split by domain responsibility |
| 8 | Use global mutable state | Untestable, race-prone; inject dependencies |
| 9 | Write getters and setters for struct fields | Go has no property syntax; export the field directly |
| 10 | Use ORM for complex queries | ORMs generate unpredictable SQL; use sqlc or raw SQL |
| 11 | Nest packages more than 3 levels deep | Flat is better; `internal/storage/postgres/` is max |
| 12 | Return `error` and a boolean for the same condition | Choose one; errors carry more information |
| 13 | Use channels when a mutex suffices | Channels are for communication; mutexes for state |
| 14 | Over-abstract before you feel the pain | Premature abstraction is the root of complexity |
| 15 | Ignore `go vet` or linter warnings | They catch real bugs: shadow variables, format verbs |

### Quality Gates for Go PRs

| Gate | Command | Threshold |
|------|---------|-----------|
| Compilation | `go build ./...` | Zero errors |
| Unit tests | `go test -race -count=1 ./...` | All pass |
| Lint | `golangci-lint run ./...` | Zero issues |
| Coverage | `go test -coverprofile=c.out ./...` | >= 80% |
| Vulnerability scan | `govulncheck ./...` | Zero high/critical |
| Integration tests | `go test -tags=integration ./...` | All pass |
| Binary size | `ls -la bin/` | < 30MB (typical API) |

### Agent Instructions — Go Developer

When implementing a Go spec, the `go-developer` agent follows:

1. Read the spec and identify affected packages
2. Write or update domain types in `internal/model/` FIRST
3. Define interfaces at the consumer (`internal/service/`)
4. Implement storage layer (sqlc queries if DB, then `go generate`)
5. Implement service layer with business logic
6. Implement handler layer (thin HTTP adapter)
7. Write table-driven tests for service layer
8. Write integration tests for storage layer (testcontainers)
9. Run all quality gates before marking done
10. Every error path must be tested — not just the happy path
